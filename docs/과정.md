1. 과제명Bobcat T590 운영 매뉴얼 기반 AWS 서버리스 RAG 챗봇 MVP 구축2. 과제 정의'Bobcat T590 운영 및 유지보수 매뉴얼' 1권(PDF)을 대상으로, AWS 서버리스 아키텍처(S3, Lambda, Bedrock, OpenSearch Serverless) 기반의 RAG(증강 검색) 파이프라인을 2주 내에 구축합니다.본 과제는 단순 텍스트 검색을 넘어, **unstructured.io**를 활용한 'Layout-Aware Parsing' 전략을 적용하여, 매뉴얼 내 테이블(Table) 데이터(예: 정비 주기표)까지 정확하게 검색하고, 사용자의 자연어 질문에 **출처(페이지 번호)**를 포함한 답변을 제공하는 MVP(최소 기능 제품) 개발을 목표로 합니다.3. 성공의 정의본 MVP의 성공은 다음 세 가지 기준을 모두 충족하는 것으로 정의합니다.기술 검증 (End-to-End): 설계된 AWS 서버리스 아키텍처(S3 → Lambda 1 → OpenSearch → Lambda 2 → Bedrock → API Gateway → UI)가 완벽하게 연동되어 End-to-End로 작동한다.정확성 검증 (Use-Case): 정의된 4대 핵심 테스트 시나리오(TC-001~004)에 대해 80% 이상 정확한 답변과 근거를 제시한다.가치 검증 (User Feedback): 현장 테스터(1~2명)가 MVP 사용 후, "기존에 매뉴얼을 직접 찾는 것보다 챗봇을 이용하는 것이 더 빠르고 유용하다"고 주관적으로 동의한다.4. Streamlit on EC2로 신속하게 구현하고, 사용자 인증(Cognito)이나 대화형 맥락(Chat History) 기능은 MVP 범위에서 제외하여 핵심 RAG 파이프라인 구축에 집중합니다.6. 선정 사유 및 기대 효과선정 사유 (데이터): 'Bobcat T590 O&M 매뉴얼'은 '안전', '유지보수', '작동법' 섹션이 명확하고, '테이블(표)' 형태의 데이터를 다수 포함하고 있어, 실제 제조업 현장의 복잡한 문서 특성을 대표하며 RAG 성능을 테스트하기에 가장 적합합니다.선정 사유 (기술): AWS Bedrock, OpenSearch Serverless 등 최신 AWS GenAI 스택을 활용한 서버리스 파이프라인 구축 역량을 검증하기에 최적의 과제입니다.기대 효과 (MVP 가설 검증):현장 작업자의 **정보 검색 시간(Time-to-Information)**을 (기존: 수동 검색 5~10분) → (챗봇: 1분 이내)로 획기적으로 단축시킬 수 있다는 핵심 가설을 검증합니다.기대 효과 (확장성):MVP 성공 시, 다른 설비 매뉴얼, 수리 이력, 안전 규정(MSDS) 등으로 데이터를 확장하여 **'전사적 제조업 인사이트 챗봇'**으로 발전시킬 수 있는 기술적 기반(Baseline)을 확보합니다.7. 공통 질문1) 어떤 업무에 어떤 형태로 적용 가능한가?MVP가 성공적으로 검증되면 다음과 같은 업무에 즉시 적용(확장)할 수 있습니다.현장 유지보수 (O&M): 작업자가 모바일 기기로 "엔진 오일 점검 주기 알려줘"(TC-002) 또는 "유압 필터 교체 방법 보여줘"라고 질문하면, 챗봇이 즉시 매뉴얼의 정확한 절차와 페이지를 제시합니다.긴급 장애 대응 (Troubleshooting): 설비 알람 발생 시, 작업자가 "Alarm 101" 또는 "시동이 안 걸려"(TC-001)라고 질문하면, 챗봇이 매뉴얼의 '문제 해결' 섹션을 기반으로 1차 원인과 조치 사항을 가이드합니다.안전 및 신입사원 교육 (OJT): 신입 작업자가 "T590의 안전 장비 목록이 뭐야?"(TC-001) 또는 "작업 전 안전 점검 항목 알려줘"라고 질문하면, 챗봇이 정확한 안전 규정을 즉시 제공하여 OJT 교육 자료로 활용됩니다.2) 보안을 위해 어떤 조치를 해야하는가?본 MVP 아키텍처에서는 다음 3가지 핵심 보안 조치를 적용합니다.IAM 최소 권한 원칙 (가장 중요):Lambda-Embed-Role(Lambda 1)은 S3 Read, Bedrock Invoke, OpenSearch Write 권한만 부여합니다.Lambda-Query-Role(Lambda 2)은 Bedrock Invoke, OpenSearch Read 권한만 부여합니다.두 Lambda가 서로의 리소스나 불필요한 AWS 서비스에 접근할 수 없도록 엄격히 분리합니다.민감 정보 관리:OpenSearch 엔드포인트 URL, S3 버킷명 등 민감하거나 변경될 수 있는 정보는 Lambda 코드에 하드코딩하지 않고, Lambda 환경 변수로 주입하여 관리합니다.네트워크 접근 제어:API Gateway CORS: UI(Streamlit on EC2)가 배포된 특정 도메인(또는 IP)에서만 API Gateway를 호출할 수 있도록 CORS(Cross-Origin Resource Sharing) 설정을 적용합니다.(MVP 이후) 향후 실제 서비스 시, API Gateway에 Amazon Cognito 사용자 인증을 추가하여 허가된 사용자만 챗봇을 사용하도록 통제합니다.3) 실제 데이터 연동을 위해 어떤걸 고려해야하는가?MVP(T590 매뉴얼 1권) 이후, 실제 데이터(수십, 수백 종의 매뉴얼)를 연동(확장)하기 위해서는 다음 4가지를 반드시 고려해야 합니다.데이터 포맷의 비일관성 (가장 큰 문제):'Bobcat T590' 매뉴얼과 'Develon DX225' 매뉴얼은 목차 구조, 테이블 형식, 용어 사용이 모두 다릅니다.해결책: unstructured.io를 사용하더라도, 문서 유형(템플릿)별로 청킹(Chunking) 전략을 커스터마이징하는 전처리 로직이 Lambda 1에 추가되어야 합니다.스캔본(Image-based) PDF 처리:현재 unstructured.io는 텍스트 기반 PDF만 처리할 수 있습니다. 현장의 오래된 매뉴얼은 스캔본(단순 이미지)일 수 있습니다.해결책: Lambda 1 파이프라인 전처리 단계에 **Amazon Textract (OCR)**를 도입하여, 스캔본 PDF에서 텍스트와 테이블을 추출하는 로직을 추가해야 합니다.증분 임베딩 (Incremental Embedding):매뉴얼이 개정될 때마다 수백 권의 문서를 처음부터 다시 임베딩하는 것은 비효율적입니다.해결책: 개정된 문서만 식별하여 OpenSearch의 기존 문서를 업데이트(Update)하거나 삭제(Delete)할 수 있는 문서 관리 및 증분 임베딩 로직을 Lambda 1에 구현해야 합니다.다국어 및 다중 모델 지원:T590 영문 매뉴얼 외에 한글 매뉴얼, 혹은 DX225 매뉴얼을 동시에 검색해야 할 수 있습니다.해결책: Bedrock Titan Embedding은 다국어를 지원하므로 큰 문제는 없으나, OpenSearch의 메타데이터에 '모델명', '언어' 등을 추가하여, "DX225 한글 매뉴얼에서 찾아줘"와 같은 필터링 검색이 가능하도록 인덱스 전략을 수립해야 합니다.
