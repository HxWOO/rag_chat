1. 2주 MVP에 가장 중요한 '속도' 확보
만약 우리가 'Serverless'가 아닌 일반 OpenSearch 클러스터나 EC2에 직접 DB를 설치했다면, 2주 중 최소 3~4일은 다음과 같은 인프라 작업에 낭비했을 것입니다.

어떤 EC2 인스턴스 타입을 쓸지 용량 산정

클러스터 노드(Node) 개수 설정

네트워크 및 보안 그룹 설정

OS 및 DB 패치, 안정화 작업

하지만 OpenSearch Serverless를 쓰면, 우리는 이 모든 과정을 건너뛰고 **버튼 하나만 눌러 즉시 '벡터 DB 엔드포인트'**를 얻을 수 있습니다. 우리는 '인프라 관리자'가 아닌 'RAG 개발자'로 2주를 보낼 수 있습니다.

2. '자동 스케일링'으로 인한 개발 편의성
RAG 파이프라인은 리소스를 예측하기 어렵습니다. 특히 Lambda 1(임베딩 함수)이 처음 실행될 때, T590 매뉴얼 전체를 한꺼번에 임베딩하면서 DB에 엄청난 쓰기(Write) 작업이 폭주합니다.

일반 DB: 우리가 미리 설정한 용량(예: t3.medium)을 초과하면 DB가 멈추거나 에러가 발생합니다.

Serverless: 이 부하를 자동으로 감지하고 즉시 확장하여 모든 작업을 문제없이 처리합니다. 우리는 성능 저하 걱정 없이 Lambda 로직 개발에만 집중하면 됩니다.

3. RAG에 최적화된 '벡터 검색' 성능
OpenSearch는 태생부터 '검색 엔진'입니다. 따라서 RAG의 핵심인 k-NN (유사도 벡터 검색) 성능이 매우 뛰어납니다. Lambda 2가 "엔진 오일" 질문 벡터를 보냈을 때, OpenSearch는 T590 매뉴얼에서 가장 관련성 높은 '테이블 청크'를 빠르고 정확하게 찾아냅니다.

4. 강력한 '하이브리드 검색' 지원 (미래 확장성)
이것이 OpenSearch의 핵심 강점입니다. MVP 이후 챗봇 퀄리티를 높일 때, OpenSearch는 **'벡터 검색'**과 **'메타데이터 필터링'**을 결합할 수 있습니다.

예를 들어, "T590 매뉴얼의 '안전' 섹션에서 '오일' 관련 내용 찾아줘"라는 복잡한 질문이 가능해집니다.

1단계: '섹션' 메타데이터가 '안전'인 문서만 필터링합니다.

2단계: 그 결과 내에서 '오일'과 가장 유사한 벡터를 검색합니다.

이는 단순 벡터 검색보다 훨씬 더 정확한 답변을 제공하며, Serverless 버전에서도 이 강력한 기능을 동일하게 지원합니다.
